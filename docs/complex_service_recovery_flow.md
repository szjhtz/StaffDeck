# 会员权益补偿与仓配优先改派流程

## 业务背景

当会员用户反馈“权益没有到账”“承诺送达时间没有达成”“希望补偿或改派”时，客服需要先核实订单、用户身份、权益发放状态和仓配状态，再决定是否补发权益、提交配送改派或发起补偿复核。

这类问题经常不是单一诉求。用户可能在一句话里同时表达多个目标，例如：“我是黑金会员，昨天买了 A3，订单 A12345，说好今天到，会员券也没到账，帮我处理一下。”系统应将该请求拆成订单核实、权益核实、配送评估、补偿判断几个连续任务，并在已获取的信息充足时直接推进，不要重复追问已经给出的字段。

本流程适用于会员权益、配送承诺、补偿申请的组合处理。不适用于单纯商品咨询、单纯购买、单纯退货退款；如果用户在其他流程中追加权益或配送承诺问题，应优先判断是否需要挂起原流程，再处理当前更紧急的售后诉求。

## 服务目标

1. 识别用户是否在咨询权益到账、配送时效、补偿、改派或组合问题。
2. 从用户当前消息、历史对话和用户记忆中提取订单、身份、会员等级、权益类型、配送要求等信息。
3. 核实订单归属和订单状态，不基于用户描述直接承诺补发或赔偿。
4. 通过权益核对接口判断应发权益、实发权益和差异原因。
5. 通过仓配评估接口判断是否可以改仓、换承运商、升优先级或拆单。
6. 涉及权益补发、配送改派、补偿复核等会改变用户权益或订单履约状态的动作前，必须获得用户确认。
7. 最终回复必须给出明确结果、失败原因或下一步人工处理方式，不能以“请稍候”“正在处理”作为结束。

## 信息收集

优先从用户当前消息中一次性抽取多个字段。已满足的信息不得重复追问。

必要信息包括：

- `user_id`：用户 ID；如果用户只提供姓名、手机号或会员昵称，可先记录为身份线索。
- `order_id`：订单号；如果缺失，可收集商品名、下单时间、收货手机号后四位辅助定位。
- `member_level`：会员等级，例如普通会员、黄金会员、黑金会员。
- `benefit_type`：用户关心的权益类型，例如积分、优惠券、等级礼包、售后优先资格。
- `benefit_campaign_id`：活动批次或权益活动名称；用户不知道时可留空。
- `delivery_expectation`：用户期望的配送结果，例如今天到、换快递、加急、改地址。
- `target_address`：涉及改派或改地址时使用。
- `user_confirmation`：用户对补发、改派、补偿复核等动作的明确确认。

如果用户一次提供多个字段，例如“我是 hm，订单 A12345，黑金会员，券没到账还没发货”，应同时记录 `user_id`、`order_id`、`member_level`、`benefit_type` 和配送问题。

## 流程步骤

### Step 1：识别组合诉求

判断用户是否只提出一个问题，还是同时包含权益、配送和补偿。组合诉求不应只处理最后一个关键词。若用户明确表达紧急售后问题，可挂起当前未完成的购买或咨询流程；若用户只是补充同一订单的信息，应继续当前流程并更新上下文。

客服回复应先说明处理顺序，例如先核实订单，再核实权益和配送，最后给出可执行方案。

### Step 2：核实订单基础状态

当 `order_id` 已存在时，查询订单状态，确认订单是否存在、是否属于当前用户、支付状态、发货状态、商品明细、下单时间、承诺送达时间，以及是否已经存在售后或补偿记录。

如果订单号缺失，先用最少问题补齐定位条件。若订单不存在或不属于当前用户，不展示敏感信息，只说明无法核实并请求用户提供正确订单信息。

### Step 3：核对会员权益应发与实发差异

权益问题不能只看订单详情里的单个字段。权益核对需要访问会员权益核对服务，用订单和会员信息拉取应发权益、实发权益、发放任务状态和差异原因。

权益核对服务的接口信息如下：

- 请求方式：`POST`
- 地址：`http://127.0.0.1:8000/api/mock/member/benefit-reconcile`
- 请求参数：
  - `user_id`：用户 ID 或会员身份标识，必填。
  - `order_id`：订单号，必填。
  - `member_level`：会员等级，可选。
  - `benefit_type`：权益类型，可选，例如 coupon、points、gift。
  - `benefit_campaign_id`：活动批次，可选。
- 样例参数：

```json
{
  "user_id": "user_demo",
  "order_id": "A12345",
  "member_level": "black",
  "benefit_type": "coupon",
  "benefit_campaign_id": "vip_2026_midyear"
}
```

- 返回字段：
  - `found`：是否找到权益核对记录。
  - `eligible`：是否符合权益发放条件。
  - `expected_benefits`：应发权益列表。
  - `delivered_benefits`：已发权益列表。
  - `missing_benefits`：缺失权益列表。
  - `difference_reason`：差异原因。
  - `recommended_action`：建议处理方式，例如补发、提示领取、人工复核、无需处理。
  - `can_auto_compensate`：是否允许自动补发或自动补偿。

处理规则：

- 如果 `eligible=true` 且存在 `missing_benefits`，说明差异原因，并在补发前请求用户确认。
- 如果权益已经发放但用户未领取，告知领取方式，不重复补发。
- 如果用户等级或活动门槛不满足，说明不满足原因。
- 如果 `recommended_action=manual_review`，提交人工复核并说明预计反馈时间。
- 如果接口不可用或返回结果不足以确认差异，不能编造补偿结果，应说明需要人工处理。

### Step 4：评估配送承诺与改派方案

配送问题需要核对订单履约状态、承诺送达时间、仓库状态、承运商能力和可拦截窗口。用户要求“今天必须到”“换个快递”“帮我加急”时，不应直接承诺，而是先计算可行方案。

仓配评估服务的接口信息如下：

- 请求方式：`POST`
- 地址：`http://127.0.0.1:8000/api/mock/fulfillment/reroute-plan`
- 请求参数：
  - `order_id`：订单号，必填。
  - `user_id`：用户 ID，可选。
  - `target_address`：目标地址或地址变更说明，可选。
  - `expected_delivery_time`：用户期望送达时间，可选。
  - `allow_split_package`：是否接受拆单，可选。
  - `blocked_carriers`：用户不接受的承运商列表，可选。
  - `member_level`：会员等级，可选。
- 样例参数：

```json
{
  "order_id": "A12345",
  "user_id": "user_demo",
  "target_address": "上海市浦东新区示例路 88 号",
  "expected_delivery_time": "2026-06-04T20:00:00+08:00",
  "allow_split_package": true,
  "blocked_carriers": ["slow_express"],
  "member_level": "black"
}
```

- 返回字段：
  - `reroutable`：是否可改派。
  - `current_route`：当前履约线路。
  - `plans`：候选方案列表，包含方案类型、预计送达时间、风险、费用和承运商。
  - `recommended_plan_id`：推荐方案 ID。
  - `requires_confirmation`：是否需要用户确认。
  - `failure_reason`：不可改派原因。

处理规则：

- 如果不可改派，说明原因，并给出跟催、补偿复核或人工处理选项。
- 如果存在多个方案，展示最稳妥和最快方案，让用户确认。
- 如果改派会导致拆单、额外费用、地址风险或时效不确定，必须提示用户。
- 用户确认后，再提交改派或创建人工工单。

### Step 5：生成补偿判断

补偿判断必须基于订单、权益和配送核实结果。权益应发未发时优先补发权益；配送承诺未达成且非用户原因时，可进入延迟补偿复核；权益和配送都异常时，可以合并说明处理依据，但不得承诺超出规则的赔偿。

如果用户要求现金赔偿、额外赠品或超规则补偿，应说明可以提交人工复核，不直接承诺。

### Step 6：请求用户确认

以下动作前必须确认：

- 补发优惠券、积分或礼包。
- 提交配送改派。
- 提交补偿复核。
- 创建人工工单并附带用户身份、订单、地址等信息。

确认问题应明确对象和结果，例如：

- “是否为订单 A12345 补发缺失的会员券？”
- “是否按推荐方案改为同城仓优先配送？可能拆单，预计今晚 20:00 前送达。”
- “是否提交延迟补偿复核？预计 24 小时内反馈。”

如果当前只有一个待确认事项，用户回复“可以”“行”“帮我弄”可视为确认。如果同时存在多个待确认事项，需要明确用户确认的是哪一个。

### Step 7：执行并闭环

执行完成后，回复必须包含：

- 已核实的订单号或脱敏订单号。
- 权益处理结果：已补发、已领取、无需补发、需复核、失败原因。
- 配送处理结果：已改派、不可改派、已创建工单、预计时效。
- 补偿处理结果：已提交、已拒绝、需人工复核、预计反馈时间。
- 用户是否还需要进一步操作。

可见回复必须基于实际核实结果组织，不要使用“马上到账”“一定成功”“请稍候后续反馈”作为最终结果。

## 异常处理

### 信息不足

用户只说“给我补偿”时，先询问订单或补偿原因。用户只说“券没发”时，先补齐订单或身份信息。

### 多订单

用户一次提供多个订单时，逐一核实并分别回复结果。如果接口或执行轮数不足以全部处理，说明已完成部分和剩余部分的人工处理方式。

### 接口失败

权益核对或仓配评估接口失败时，应保留已核实的订单结果，说明自动核实失败原因，并转人工或创建复核任务。接口失败不能被解释为用户不满足权益或不可改派。

### 用户追加新诉求

若用户在处理权益或配送时追加购买、退款、取消订单等新诉求，应先判断当前是否处于等待确认。若当前需要确认，先完成确认；若当前流程已闭环，可继续处理新诉求。

### 情绪激烈

先承接情绪，再核实事实。不能因为用户态度强烈就跳过订单核实、接口核实或用户确认。
